# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

Raku::Playground is a Swift-Playgrounds-inspired page that runs Raku entirely in the browser: a CodeMirror 6 editor, a Run button, and an output pane, powered by Rakudo compiled to JavaScript. It is a pure static site under `docs/` (GitHub Pages-friendly, no backend), wrapped in an App::Mi6-managed Raku distribution.

## Commands

```sh
# Serve the playground locally (zero-dependency Raku static server)
raku tools/serve.raku
# then open http://localhost:8000 (first load takes ~1-2 min: perl6.js is 77MB)

# Run the Raku test suite (same as CI)
prove6 -I. t

# Run a single test file
raku -I. t/01-basic.rakutest

# Browser tests (Playwright; starts tools/serve.raku if :8000 is free)
npm install && npx playwright install chromium   # once
npx playwright test              # all specs (~2 min; loads the 77MB runtime per spec file)
npx playwright test tests/world.spec.js   # one spec

# Regenerate README.md from the pod in the module, update META6.json (App::Mi6)
mi6 build
```

CI: `.github/workflows/test.yml` runs `prove6 -I. t` on Linux/macOS/Windows against the latest Raku; `.github/workflows/playwright.yml` runs the browser suite on every push and PR (bundled Chromium everywhere — the system Chrome channel hangs at shutdown; teardowns are timeout-guarded). The specs share one page per file (`serial` mode) to avoid reloading the 77MB runtime per test; each spec **launches its own browser** in `beforeAll` (`chromium.launch()`, not the worker-scoped `browser` fixture) and its `afterAll` must close the page **and the browser** (skipping the browser close makes workers hang for 5 minutes at shutdown and fail the run). The explicit launch is load-bearing: with more spec files than workers, a worker runs several files, and the worker-scoped `browser` fixture is **reused** across them — so a file that closed the fixture browser in its `afterAll` would leave the next file on that worker with a closed browser (`browser.newPage()` → "Target page, context or browser has been closed"). Each file owning its own browser decouples the lifecycle. **Keep `workers: 3`** (up to 3 spec files' browsers in flight at once): the runtime now runs in a Web Worker, so each browser has two hot threads, and 3 browsers oversubscribe CI's 2 cores (~5× slower runs) — hence the generous `timeout` and interval-based (not frame-rate) polling in `waitIdle`. Do **not** drop to `workers: 1` — every spec file would serialize onto one worker and run ~5× longer wall-clock.

## Architecture

The playground is framework-free vanilla JS in `docs/`:

**The runtime runs in a Web Worker** (worker-only architecture — main thread never runs Rakudo). This means a run never freezes the UI, and runaway loops are cancellable. No `SharedArrayBuffer` is used (GitHub Pages can't set COOP/COEP), so every Raku↔JS interaction is either fire-and-forget or answered inside the worker — see the puzzle/grammars notes below.

- `docs/perl6.js` — Rakudo compiled via its JavaScript backend, bundled with Parcel (~77MB, copied verbatim from `../MemoizedDOM/examples/todo-webperl/`). It sets `evalP6(code)` on its global. **Verified it boots under a Worker's `self`** with only `window` aliased to `self` (no `document` needed at init). Raku code can reach JS via `EVAL :lang<JavaScript>` — in the worker that reaches worker globals (`self.PG`, `self.NQP_STDOUT`), not the DOM.
- `docs/raku-worker.js` — the Web Worker. Aliases `window = self`, sets `self.NQP_STDOUT` **before** loading perl6.js, streams+compiles perl6.js (fetch with progress → Blob → `importScripts`; `importScripts('world-sim.js')` for the sim), and owns the message protocol. Exposes `self.PG = {command, query}` over a worker-local `WorldSim`. Wraps `console.error` for stderr (Rakudo has no NQP_STDERR hook) and forwards it only during a run + a short grace window. `exit()` throws an `Exit`-named JS exception → clean end-of-run. **Single-file build**: when `self.PERL6_EMBEDDED` is set (perl6.js concatenated into the worker blob), it polls for `evalP6` instead of fetching.
- `docs/world-sim.js` — the DOM-free puzzle **simulation** (`WorldSim`: `command`/`query`/`walkable`/`result`, the 1000-command runaway guard, direction tables). A classic script (attaches `WorldSim` to `self`) so the worker can `importScripts` it. `docs/world.js` keeps only render/playback and replays streamed commands.
- `docs/elevator-engine.js` — the **Elevator saga** (`type: "elevator"`, a Raku port of [Elevator Saga](https://play.elevatorsaga.com/)). Unlike the puzzle sim, the **whole simulation runs in Raku inside the worker, in one `evalP6` run**, so the game is genuinely event-driven: the player writes `sub init(@elevators, @floors)` / `sub update($dt, …)` and registers closures with `$e.on('idle'|'floor-button-pressed'|'stopped-at-floor'|'passing-floor', …)` / `$f.on('up-button-pressed'|'down-button-pressed', …)`, and a Raku-hosted engine (exported as the `ELEVATOR_ENGINE` string) drives time and fires those handlers. No `EVAL`/JS bridge and no JS→Raku callbacks — the engine ships **presentation events** on the reliable stdout channel with a `@@EV@@` sentinel (mirrors grammars' `@@RX@@`; ASCII `type|t|arg…` fields, one flush line per sim tick): `raku-worker.js` parses them, tallies an `ElevatorPresenter` result, and forwards each as a `command`. `elevator-engine.js` also exports `class Building` (main-thread renderer/playback): it buffers the events and `playAll(speed)` replays them on a wall-clock-scaled timeline (cars slide via a CSS `bottom` transition, doors/board/alight/HUD update). **buildSource composes `[userSrc, config, ELEVATOR_ENGINE]`** — user code first so their compile-error line numbers stay right; per-level config is Raku constants derived from the level's JS fields (`floors`/`elevators`/`capacity`/`seed`/`spawns`/`budget`). All engine classes are `my`-scoped; spawns come from a **seeded LCG in Raku** so every level's `solution` is deterministic for headless verification.
- `docs/snake-engine.js` — the **Snake Arena saga** (`type: "snake"`, a Battlesnake-style game). Same architecture as elevator (whole sim runs in Raku in the worker in one `evalP6`): the player writes `sub move($you, $board)` returning a direction (`'up'|'down'|'left'|'right'`), and the Raku engine (exported as `SNAKE_ENGINE`) ticks time, advances the snake(s), resolves collisions and streams **presentation events** on stdout with a `@@SN@@` sentinel (its own ASCII `type|t|arg…` grammar; `k` = a snake's body this tick, variable-length coords). `raku-worker.js` parses them, tallies a `SnakePresenter` (honest length/food/death from the trusted event stream) and forwards each as a `command`. `snake-engine.js` also exports `class Arena` (main-thread renderer/playback into `#arena`): a CSS-grid board, snake segments and food; `playAll(speed)` replays on a tick-scaled timeline. **buildSource composes `[userSrc, snakeConfig(level), SNAKE_ENGINE]`** (user code first). Per-level config is Raku constants from the level's JS fields (`W`/`H`/`seed`/`ticks`/`food`/`start`/`opponent`/`budget`); coordinates are `(0,0)` bottom-left, up = +y; all classes `my`-scoped; food + the built-in rival come from a **seeded LCG in Raku** so every `solution` is deterministic. The API (`$board.nearest-food`/`step-toward`/`neighbor`/`in-bounds`/`occupied`, `$you.head`/`body`/`length`, `Pos.dist-to`) is **method-based** — user `move` compiles before the engine classes, so navigation must be late-bound method dispatch, not free subs. The capstone rival has a wake-up `delay` and length goals track *max* length, so a death after reaching the target still wins — both keep the headless `solution` robust to RNG.
- `docs/raku-runtime.js` — main-side **async client** of the worker (spawns it, relays progress, `postMessage`s runs). Public surface: `state`, `onStateChange`, `onStdout`, `onStderr`, `onProgress`, `onCommand` (puzzle command stream), `onRender` (grammars highlight payload), an **async `run(source, level)`** that resolves with the puzzle result, and `cancel()` (terminate + respawn → the Stop button, plus a 30s wall-clock guard). Key constraints:
  - **NQP_STDOUT is HTML** (ansi-to-html with escapeXML: user text entity-encoded, ANSI colors as real `<span>` tags). The worker forwards it raw; the **main side decodes it via `DOMParser`** (workers have no DOMParser) — do not just strip `<`/`>`, that mangles the span tags Rakudo uses for compile errors. **`die` messages and compile errors arrive through NQP_STDOUT**, so they render as regular output, not the stderr color; `note`/`warn` arrive via the wrapped `console.error` (stderr color).
  - **Cross-eval state does not persist**: each `evalP6` call compiles a fresh unit; lexicals from a previous run are not visible in the next (verified).
  - States: `uninitialized → loading → ready → running → ready` (or `error`). `run` is async — `playground.js` `await`s it, then animates.
- `docs/playground.js` — wires the editor, saga/level/speed **and theme** selects, Run/**Stop**/Step/Clear buttons, instructions panel, status badge, output pane, world pane, and preview pane to the runtime client. Streams `onCommand` into `world.commands`, then `playAll` animates. Owns the editor keymap (Cmd/Ctrl-Enter to Run; **Tab/Shift-Tab indent by 4 spaces**, Escape releases focus) and the Shiki highlighter + theme chooser (see the `codemirror.js` bullet).
- `docs/tour.js` — first-run **onboarding tour**: a self-contained (no-import) spotlight coach-mark walkthrough that dims the page, cuts a bright `box-shadow` "hole" over one pane/button at a time (`z-index:2000`, above the `#load-bar`), and floats a tooltip card (Back/Skip/Next, ⌘/Esc/arrow keys). Auto-runs once for new visitors (persisted in `localStorage` under `raku-playground-tour-seen`) — it plays **while the 77 MB runtime downloads**, so it doesn't need the runtime ready. The header **`?` button** (`#help`) replays it anytime; `startTour` is also on `window.__playground`. Steps whose target is `hidden` (e.g. `#world`/`#speed`/`#step` in free/dom mode) are skipped. Bundled into the single-file build automatically via the `playground.js` import (a bare `<script>` tag would not be). Styles live in `style.css` under the `.tour-*` section; tested by `tests/tour.spec.js` (fast — never waits for runtime idle).
- **Preview pane** (`#preview`): now used **only by grammars** (the render receiver draws `<pre>/<mark>` highlights there). Free play is text-only (the runtime has no DOM); there is no `window.PREVIEW` and no free-play DOM example anymore.
- **Puzzle world** (`docs/world.js` render/playback + `docs/world-sim.js` sim): the sim runs **in the worker**; it records each command and **streams them to the main thread to animate afterwards** (queries are answered locally & synchronously from the worker-local sim — no cross-thread round-trip). A single-line Raku prelude (`PRELUDE`, in world.js) defines the kebab-case commands `move-forward`, `turn-left`, `turn-right`, `collect-gem` and the queries `is-blocked`, `is-blocked-left`, `is-blocked-right`, `is-on-gem`, `gems-left` — **all declared as terms** (`sub term:<…>`), never plain subs: a plain sub is a listop that slurps what follows, breaking `until is-blocked { … }` (block parsed as argument) and `move-forward xx 2` (`xx` parsed as an undeclared routine). As terms they compose with `xx`, `for ^n`, and statement modifiers; the trade-off is that call syntax with parens (`move-forward()`) is not supported. All of them call `self.PG.command/query` (in the worker) via `EVAL :lang<JavaScript>`; the sim advances instantly so queries are answered mid-eval. Key constraints:
  - The prelude interpolates into EVAL strings, so it starts with `use MONKEY-SEE-NO-EVAL;`.
  - The prelude is one line, so **user compile-error line numbers are off by exactly one** in puzzle mode.
  - Queries return 1/0 and the prelude compares `== 1` (numeric coercion across the JS boundary is more reliable than JS booleans).
  - The sim's `command` throws after 1000 commands (runaway-loop guard) — the throw ends the run and the commands recorded so far still animate. A `loop { }` recording *no* commands spins the **worker** thread forever, but the **UI stays live**: the **Stop** button (`runtime.cancel` → terminate + respawn) kills it, and a 30s wall-clock guard auto-cancels a zombie run. (The old "sync eval freezes the tab" footgun is gone.)
  - After Camelia falls (moving onto water/void), further commands are silently dropped; the recorded fall animates and the run fails.
  - **Sagas** (sets of levels) live in `docs/sagas/` — one file per saga exporting `{ id, title, description, levels }` (optionally `prelude`: Raku source prepended to every level run in the saga — after the command `PRELUDE` on puzzle levels; free play never gets preludes), registered in `docs/sagas/index.js`. **To add a saga: create `docs/sagas/<id>.js` and add one import line to `index.js`** — the saga picker, per-saga progression, and verification pick it up automatically. Current sagas: `learn-raku` (16 puzzle levels), `quoting`, `data-structures`, `containers`, `types-mop`, `phasers` (5 output-checked dom levels each, per the docs.raku.org reference pages), `grammars` (8 dom levels: regexes → grammars → actions; `show-matches` computes matches in Raku and renders highlights via a worker→main render message, not the DOM), `elevator` (6 `type: "elevator"` challenge levels — the event-driven Elevator Saga port; see `elevator-engine.js`), `snake` (6 `type: "snake"` challenge levels — the Battlesnake-style Snake Arena; see `snake-engine.js`), `gem-rush` (3 challenge puzzle levels). **DOM-manipulating sagas aren't supported** (no DOM in the worker) — the former `memoized-dom` saga was removed; a new saga must be puzzle, output-checked, or use the grammars-style render-message pattern.
  - **Four level types.** Default = puzzle world. `type: "dom"` levels use the output pane and/or the preview pane: the saga's `prelude` is prepended to the run, and success is `check(previewEl, { output })` → `{ success, message? }` (`output` is the output pane's text; `previewEl` holds anything a render message drew, e.g. grammars' `<mark>`s). Result shows in `#dom-banner`. Output-only sagas ignore `previewEl` and just read `output`. `type: "elevator"` levels render into `#building` (a stage beside `#world`/`#preview`): the level carries `floors`/`elevators`/`capacity`/`seed`/`spawns` and a `budget` `{ transport, time, maxMoves?, maxWait? }` win condition (`goal` stays a display string), success is computed by the worker's `ElevatorPresenter` from the event tally — no `check` function. `applyMode("elevator")` keeps the speed/step controls; see `elevator-engine.js`. `type: "snake"` levels render into `#arena` (its own stage): the level carries `W`/`H`/`seed`/`ticks`/`food`/`start`/optional `opponent` and a `budget` `{ survive | food | length }` win condition, computed by the worker's `SnakePresenter` — no `check` function. `applyMode("snake")` also keeps speed/step; see `snake-engine.js`. Adding an elevator/snake-style stage means a new `#…` div in `index.html`, a mode branch in `playground.js` (`applyMode` + the `setLevel` selector + `buildSource`), and a worker sentinel/presenter — the three modes are hardcoded, there is no generic custom-stage hook.
  - **rakudo.js limitation:** the sequence operator `...` freezes the tab with callable generators (`* + *`, two-arg blocks) or a `*` endpoint — only deduced rules with concrete endpoints work; levels must avoid the famous lazy-Fibonacci spelling. Also `<3/4>.^name` answers `Rat`, not `RatStr`, and `.^add_method` needs a *named* method (`my method greet {…}`) for `.^methods` to report its name.
  - **Raku↔JS interop rules learned the hard way:** passing Raku strings as arguments to arbitrary JS functions (via `EVAL`-returned callables) hands the JS side a useless wrapper. In the worker there's no DOM to route through, so **reliable channels are stdout (`say`) and integer/EVAL-interpolation**. grammars uses this: `show-matches` emits an **ASCII-sentinel stdout line** (`@@RX@@<codepoints>@<from,to;…>`) that the worker intercepts (in `NQP_STDOUT`) and turns into a render message — the marker/delimiter must be ASCII because NQP_STDOUT HTML-encodes non-ASCII (e.g. `§` → `&sect;`). And a bare statement returning a `WrappedJSObject` dies with "No such method 'sink'" — assign JS-returning calls to a throwaway (`my $ = …`).
  - Explain paragraphs support `` `code` ``, `**bold**` and `*italic*` inline markers.
  - **Check design principle:** verify the essence (the right values/elements present — `/\b42\b/`, `includes(name)`), never incidental formatting (exact separators, exact phrasing); when a check must be specific, the level's goal/steps must say so explicitly, and failure `message`s state exactly what's missing.
  - **Rakudo.js gotcha discovered here: `our`-scoped symbols persist across `evalP6` calls** (lexicals don't). A plain `class Foo` re-declared on the next Run dies with "Redeclaration of symbol" — all classes in dom-level code and preludes must be **`my`-scoped** (`my class`, `my role`).
  - A saga `prelude` that is single-lined by the saga file at load (e.g. grammars') relies on a `}` only terminating a statement at end-of-line, so the join appends `;` to block-final lines; keep `if … } else {` on one physical source line there.
  - Level format: `{ name, goal, steps[], explain[], grid[], start: {x,y,facing}, starter, hint, solution }`; grid legend `#` path, `G` gem, `W` rock (bump), `~` water / ` ` void (fall). `explain` paragraphs render in the instructions panel with `` `backticks` `` becoming `<code>`. **`solution` is load-bearing**: the headless verification runs every level's solution in every saga and requires the success banner — keep it correct when editing levels.
  - Learn Raku teaches syntax progressively (statements → `xx` → `for`/ranges → `until`/`while` → conditionals → subs → `elsif`/`else` → pointy blocks → `repeat` → truthiness via the numeric `gems-left` query → `loop`/`next`/`last` → `given`/`when` → named parameters → `with`/`//`/`?? !!` → recap). Topics aligned with docs.raku.org/language/control.
  - **Progression is locked per saga**: a level is selectable only when all previous ones in its saga are completed; completion is stored in localStorage under `raku-playground-progress:<sagaId>` (a legacy un-suffixed key is migrated to `learn-raku` on load); the success banner offers a "Next level →" button. Free play is always available.
  - The board is pseudo-isometric CSS driven by `--rotX`/`--rotZ` custom properties on `#world` (default 55°/45°); **click-drag rotates it** (pointer handlers in playground.js). Sprites counter-rotate via `.upright` using `calc()` on the same variables — change angles only through the variables.
- **Single-file build**: `node tools/build-single.mjs` (esbuild on PATH or `ESBUILD=/path/to/esbuild`) produces `dist/raku-playground.html` (~75MB) — app JS bundled, CSS inlined. Because a `file://` page can't load a separate worker file, the **whole worker** (flag + `world-sim.js` + `raku-worker.js` + `perl6.js`) is emitted as one `<script type="text/plain" id="worker-src">` block; `raku-runtime.js` `spawn()` detects it and builds a **Blob-URL Worker** (the worker sees `PERL6_EMBEDDED` and polls for `evalP6`). Works from `file://`. `dist/` is gitignored.
- `docs/vendor/codemirror.js` — committed one-time esbuild bundle of CodeMirror 6 **plus a bundled, offline [Shiki](https://shiki.style) highlighter**. Highlighting uses Shiki's **real `raku` TextMate grammar** (accurate — sigils, twigils, `q//`, phasers, grammars), not the old legacy-Perl approximation, wired into CM6 via the **`codemirror-shiki`** extension. Shiki runs on the **JavaScript regex engine** (`shiki/engine/javascript`, `forgiving: true`) — no oniguruma WASM and no CDN, so everything works on GitHub Pages **and** in the `file://` single-file build. The bundle also ships ~6 Shiki **themes** (One Dark Pro default, Dracula, Nord, GitHub Dark, Catppuccin, Vitesse) exported as a `THEMES` map for the editor's theme chooser. Regenerate from the committed **`entry.js`** with:
  ```sh
  npm install codemirror @codemirror/state @codemirror/view \
      shiki @shikijs/core @shikijs/langs @shikijs/themes @shikijs/engine-javascript codemirror-shiki esbuild
  npx esbuild entry.js --bundle --format=esm --minify --outfile=docs/vendor/codemirror.js
  ```
  `entry.js` re-exports `EditorView, basicSetup, keymap, Compartment` plus `createHighlighterCore, createJavaScriptRegexEngine, shikiExt (codemirror-shiki default), synchronousHighlightEffect, rakuLang (@shikijs/langs/raku)` and the `THEMES` object. **Themes are imported statically** so esbuild inlines them (no dynamic `import()` chunks → the single-file build stays one file). `playground.js` builds the highlighter once and installs it through a **theme `Compartment`**; it passes the highlighter **as a Promise** to `shikiExt` (given a Promise, codemirror-shiki defers its first `dispatch` to a microtask, so installing the extension during a compartment reconfigure can't throw "update in progress"). Theme choice persists in `localStorage` (`raku-playground-theme`). Note the accurate Raku grammar leaves the puzzle command **terms** (`move-forward`, `collect-gem`, …) the default colour — they aren't Raku keywords; that's expected, not a highlighting failure.
  - **`shikiExt` only colours tokens.** The editor **chrome** (background, gutter, line numbers, active line, cursor, selection) is themed by an `EditorView.theme(...)` built from the Shiki theme's own editor `colors` (`editor.background`, `editorLineNumber.foreground`/`activeForeground`, `editor.lineHighlightBackground`, …) — `editorTheme(id)` in `playground.js`, reconfigured into the same theme `Compartment` as `shikiExt`. Without it the gutter/line-numbers keep CodeMirror defaults and don't follow the theme.
  - **Theme switch must force a full re-highlight.** codemirror-shiki caches tokens per line and skips re-tokenising a line whose *range* is unchanged, so a theme change alone only recolours the cursor line. `applyTheme` dispatches `synchronousHighlightEffect.of(null)` alongside the compartment reconfigure to re-highlight the whole document.
- `docs/.nojekyll` — keeps GitHub Pages from running Jekyll (harmless with the Actions deployment, needed if anyone switches back to branch deployment).
- **GitHub Pages**: `.github/workflows/pages.yml` deploys `docs/` on every push to `main` (and via manual dispatch). One-time setup after the repo exists on GitHub: set Pages source to "GitHub Actions" (Settings → Pages, or `gh api repos/{owner}/{repo}/pages -X POST -f build_type=workflow`).

## Raku Distribution Conventions

- This project is managed by App::Mi6 (`dist.ini`). **Do not edit `README.md` directly** — it is generated from the pod at the bottom of `lib/Raku/Playground.rakumod` (see `[ReadmeFromPod]` in `dist.ini`). Edit the pod and run `mi6 build`.
- `META6.json` declares the distribution: new modules go under `provides`, runtime dependencies under `depends`.
- Tests are `t/*.rakutest`; version and release notes live in `Changes`; `mi6 release` manages releases. **Caveat before any release**: the dist tarball would include the 77MB `docs/perl6.js` (mi6 packages tracked files) — likely over ecosystem upload limits; exclude the site from the dist or don't release until that's resolved.
- `.precomp/` is precompilation cache output and must never be committed or edited.
